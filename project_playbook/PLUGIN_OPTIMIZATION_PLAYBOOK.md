# 插件优化复盘与方法论

本文记录 `astrbot_plugin_ani_rss` 从功能验证到接近正式版过程中的关键优化逻辑。目标不是复述使用方法，而是沉淀一套可迁移到其他 AstrBot 插件的设计参考。

## 优化历程

### 1. 先收敛业务边界

最初目标容易扩散到“搜索、预览、确认、删除、刷新、卡片展示、订阅管理”等完整控制台能力。实际落地时先确认核心闭环：

- 用户用自然语言或命令提出追番需求。
- 插件搜索 Mikan，展示番剧候选和字幕组候选。
- 用户选定字幕组后，插件直接调用 ANI-RSS API 添加订阅。
- 后续下载、更新、规则执行交给 ANI-RSS 自身处理。

因此插件不再做二次确认字幕组选择，也不暴露高风险删除工作流。其他插件也应先明确“插件负责触发外部系统的哪一步”，不要把对端系统已有职责搬进 AstrBot。

### 2. 按架构拆分职责

根目录只保留 `main.py` 作为 AstrBot loader 入口，实际职责拆到模块目录：

- `plugin/`: 配置、SQLite、缓存、偏好、运行时工具。
- `routing/`: wake prefix、短命令和命令格式解析。
- `integrations/`: 外部 API client。
- `workflows/`: 业务 workflow 编排。
- `ui/`: 卡片渲染和媒体处理。
- `scripts/`: 兼容层和调试入口。

拆分原则是“主文件做统合入口，功能文件只承担单一角色”。超过 500 行的文件优先拆分，尤其是同时承担解析、状态、IO、渲染的模块。

### 3. 命令交互从长命令改为短命令

挂起任务最初尝试过 `/anirss continue <id> 1`、`/继续...`、`/ani-xxxx 1`、裸短码等格式。最终收敛为：

```text
/ani<任务ID前后几位> <序号|确认|取消>
plana ani68d2 1
```

关键决策：

- 保留 `ani` 命名空间，避免裸 `68d2 1` 被 AstrBot 或 LLM 当成普通聊天。
- 支持前几位或后几位模糊匹配，降低用户输入量。
- 拒绝旧格式和裸短码，给出明确提示，避免多种格式长期并存。
- wake prefix 从 AstrBot 配置读取，同时兼容线上 `plana` 这类唤醒词。

其他插件的短命令也应带业务命名空间，例如 `task68d2 1`，不要只用随机短码。

### 4. 运行时状态从内存迁移到 SQLite

仅用内存存储挂起任务会导致重载后丢失流程，这是交互型插件的大问题。当前运行态统一写入 AstrBot 插件数据目录：

```text
data/plugin_data/astrbot_plugin_ani_rss/state.sqlite3
```

SQLite 承担：

- 挂起任务和过期时间。
- Mikan 搜索缓存、字幕组缓存、ANI-RSS 订阅列表缓存。
- 用户字幕组和语言偏好。
- workflow 工具结果记忆。

配套策略：

- 定时清理过期任务和缓存。
- 插件重载后仍可继续未过期任务。
- 开发阶段不兼容旧数据库字段，避免早期 schema 迁移复杂化。

其他插件只要有“多轮交互、重载后续跑、短期远端缓存、用户偏好”任一需求，就不应只依赖内存。

### 5. 卡片渲染降级为表现层能力

卡片是用户体验优化，不应影响核心业务成功率。当前渲染策略：

- 默认发送图片卡片。
- 渲染失败自动回退文本。
- `render_mode` 支持 `image` / `text` / `both`。
- Linux 环境依赖 `pylitehtml`，不可用时仍能走文本结果。
- 订阅列表按 ANI-RSS 星期分组，默认展示 20 条，图片和文字保留间距。

其他插件应遵循：渲染失败只能影响展示，不能中断业务流程。

### 6. AI 介入只做分流，不直接做副作用

自然语言入口和 LLM tool 参数不直接落到真实 API 写入，而是先进入 `ai_dispatch`：

- 解析用户意图和上下文。
- 生成受控候选分支。
- 只有高置信度单分支才进入具体 workflow。
- 低置信度或多候选时返回选项，不做写入。

同时，`search_mikan` 作为 AI 内部检索时默认只返回文本，不主动生成卡片或挂起任务；只有交互入口或 `interactive=true` 才展示候选卡片。

可迁移原则：LLM 负责理解和路由，确定性的 Python workflow 负责执行。不要让 LLM 直接决定危险操作。

### 7. 偏好学习优先用结构化数据

字幕组、语言这类偏好是强结构化选择，不需要一开始上向量库。当前策略：

- 成功添加订阅后记录当前会话常用字幕组和语言标签。
- `rank` 模式只调整候选排序，不跳过用户选择。
- `auto` 模式需要达到最小使用次数且领先足够明确，才自动选择。
- 用户没有选择靠前偏好项时，记录负反馈。
- 偏好按时间衰减，避免早期选择永久支配。

向量存储更适合自由文本别名、模糊需求和跨上下文召回；不适合替代明确枚举项的计数排序。

### 8. 借鉴 Bilibili Push 和 Livememory 的思路

从 Bilibili Push 类插件借鉴的是“意图先分流，写入后置”的做法：用户自然语言先变成受控 workflow，再进入具体 API 调用。

从 Livememory 类插件借鉴的是“工具结果记忆”：记录 workflow 的成功/失败、摘要、渲染结果和来源。它不直接改变主流程，而是用于诊断、后续智能调用优化和问题复盘。

跨插件可复用的组合是：

- `intent dispatch`: 自然语言先分流。
- `pending task`: 不确定时挂起。
- `tool result memory`: 每次执行留下可查询摘要。
- `preference store`: 高频结构化选择沉淀为偏好。

## 当前工作流矩阵

| Workflow | 用途 | 是否用户可见 | 是否有副作用 |
| --- | --- | --- | --- |
| `ai_dispatch` | 自然语言前置分流 | 是 | 否 |
| `add_mikan_subscription` | 搜索 Mikan 并添加订阅 | 是 | 选定字幕组后写入 |
| `recommend_mikan_subscription` | 按季度/关键词推荐番剧 | 是 | 选定字幕组后写入 |
| `search_mikan` | 搜索番剧或查看字幕组 RSS | 是 | 否 |
| `add_subscription` | RSS 直连添加 | 是 | 确认后写入 |
| `continue_pending` | 继续挂起任务 | 是 | 取决于挂起任务 |
| `list_subscriptions` | 查看 ANI-RSS 已启用订阅 | 是 | 否 |
| `refresh_subscription` | 刷新单个订阅 | 是 | 是 |
| `refresh_all` | 刷新全部订阅 | 是 | 是 |
| `check_status` | 连接与运行时诊断 | 否 | 否 |
| `preview_subscription` | 预览订阅结构 | 否 | 否 |

设计检查点：所有有副作用的 workflow 都应可追踪来源、可被测试、可在低置信度时转为挂起或提示。

## 可复用优化清单

### 架构清单

- 根入口只保留插件注册和统一分发。
- 每个目录有职责文档，不把所有说明塞到根 README。
- 外部 API client 与 workflow 解耦。
- UI 渲染与业务写入解耦。
- 命令解析与业务执行解耦。
- 大文件超过 500 行时优先按职责拆分。

### 配置清单

- 配置项使用中文标题和说明，减少用户理解成本。
- 只暴露真实需要用户决策的选项。
- 超时、缓存 TTL、挂起 TTL、渲染模式属于合理配置。
- 不把内部模板、无使用场景的来源、调试开关长期暴露给正式用户。
- 不在仓库写真实 API Key、密码或本地地址。

### 交互清单

- 用户选择最终候选后可以直接执行，不要为了“显得安全”增加无意义确认。
- 高风险操作才需要确认，例如删除、覆盖、批量刷新。
- 连接状态这类诊断应后台静默做，只有失败或用户主动查询时展示。
- 多轮流程必须给任务 ID，并支持短 ID 模糊匹配。
- 旧命令格式要么兼容一段时间，要么明确拒绝并提示新格式，不要悄悄进入 LLM。

### AI 调用清单

- 自然语言进入 `dispatch`，不要直连写入 handler。
- LLM tool 的泛化参数要重新解析，不信任 `workflow=xxx` 这类模糊输入。
- 内部检索默认返回文本给模型，不主动发卡片打扰用户。
- 只有交互入口才挂起任务和渲染候选卡片。
- 低置信度时返回候选分支，让用户选。

### 数据清单

- 多轮任务、缓存、偏好、工具结果使用插件数据目录持久化。
- 挂起任务和缓存必须有 TTL。
- 成功路径和失败路径都记录摘要。
- 偏好学习先用结构化计数和衰减，确有自由文本召回需求再考虑向量。
- 开发阶段可以不兼容旧 schema，正式发布后再建立迁移策略。

### 测试清单

- `py_compile` 覆盖全部 Python 文件。
- 用 AstrBot venv 做插件导入测试。
- 本地 AstrBot 启动后检查日志里插件加载和 LLM tool 注册。
- 用 webchat 发送安全命令，例如状态检查或订阅列表。
- 清理测试会话、`__pycache__` 和临时文件。
- Linux 兼容重点检查大小写路径、路径分隔符、依赖 wheel、渲染降级。

## 后续优化方向

- 增加更细的 workflow 执行指标，例如远端耗时、缓存命中、渲染耗时。
- 将工具结果记忆用于自动生成问题诊断摘要，但不直接改变副作用 workflow。
- 对自然语言别名、番名别称、用户常见口误引入轻量召回；结构化偏好仍保持 SQLite 排序。
- 对刷新类 workflow 增加更严格的批量操作提示，避免误触发。
- 发布稳定版后再设计 SQLite schema migration，不在开发期背负旧字段兼容。
- 若未来扩展搜索源，按 `integrations/source_x.py` + `workflows/source_x_*` 增加，不让 Mikan 逻辑继续膨胀。

## 给其他插件的推荐方案

如果插件也要接入外部服务并支持自然语言，建议使用以下基础架构：

```text
main.py
plugin/
  settings.py
  sqlite_store.py
  cache_store.py
  preferences.py
  tool_results.py
routing/
  commands.py
integrations/
  external_api.py
workflows/
  models.py
  dispatch.py
  parsing.py
  runner.py
  <business>.py
ui/
  rendering.py
  cards.py
project_playbook/
  <plugin>_optimization_playbook.md
```

执行顺序建议：

1. 先做最小业务闭环。
2. 再拆分模块边界。
3. 加 SQLite 持久化和 TTL。
4. 加短命令和挂起任务。
5. 加卡片渲染和文本降级。
6. 加 AI dispatch。
7. 加偏好学习和工具结果记忆。

不要反过来先做复杂记忆、向量库或视觉模板。插件稳定性的核心是清晰的副作用边界和可恢复的运行时状态。
